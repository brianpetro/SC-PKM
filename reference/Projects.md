> [!summary]
> Index of active projects (deliverables with an end) in a PARA-aligned system.

> [!When to use]
> If you cannot tell what is active right now, this note helps you define Projects as time-bound efforts so execution stays clear, even with lots in flight.

### Purpose and scope
- A Project is a time-bound outcome with a clear finish line (a deliverable, decision, or launch).
- If it has a deadline or a 'done' state, it belongs here; if it is ongoing, it probably belongs in [[Areas]].
- Prefer one hub note per project so context, decisions, and links do not scatter

### How to use this MOC
- List only active projects as wikilinks; keep the list short enough to scan in 30 seconds.
- Each project link should point to a hub with: objective, success criteria, and one current next action.
- During review, move completed or paused projects to [[Archives]] (do not delete).

### Key links
- [[Decision Logs]] keeps 'why we chose this' attached to the project, not lost in chat.
- [[Meeting Notes Workflow]] turns meetings into decisions and actions that link back to the hub.

### Maintenance and review
- Weekly: make sure every active project has a next action and a current focus.
- Monthly: close or archive projects that are not moving; reduce the active list.

### Common pitfalls
- Mixing Areas into Projects (no deadline) - move them to [[Areas]] and define a maintenance habit.
- Letting the hub become a diary - keep only what supports decisions, actions, and reuse.
