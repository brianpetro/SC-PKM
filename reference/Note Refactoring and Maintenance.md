> [!summary]
> Keep notes healthy by splitting, merging, renaming, and updating as thinking evolves.

> [!When to use]
> If notes age into clutter, this note shows when and how to split, merge, and update so the vault stays healthy, without turning it into a hobby.

### When to refactor (signals and triggers)
- The note covers multiple ideas and is hard to link.
- You cannot summarize the note in one sentence.
- The title no longer matches what you would search.

### Split, merge, and rewrite decision rules
- The note contains two claims that could be reused separately.
- One section keeps changing while the rest is stable.
- Two notes are duplicates or near-duplicates.

### Update links, aliases, and metadata safely
- If you rename a note, add an alias with the old name (see [[Note Naming Conventions]]).
- Update the "related" list only if you also add contextual sentences.
- Keep tags stable; if you change tags, consolidate in [[Tag Canon]].

### Archive and deprecate stale notes
- The note is historically useful but no longer current.
- The content is tied to a past project.
- The note is wrong and you do not want to use it again.

### Maintenance cadence and lightweight checklist
- [ ] One example and one failure mode exist (for concept/method notes).
- Daily: 2 minutes of micro-fixes (rename one note, add one alias).
- Weekly: 10-20 minutes of gardening.
