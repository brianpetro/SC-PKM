### What categorization is trying to solve
- Fast retrieval: find the right note quickly.
- Reuse: discover related notes when you are writing or deciding.
- a search problem (hard to find),

### Tagging vs foldering vs linking roles
- Links: explicit meaning and relationships ("supports", "contrasts", "example of").
- Folders: storage and lifecycle boundaries (ex: inbox vs archive).
- Tags: grouping and queues (ex: #review, #drafting).

### Minimal category set and when to expand
- Add a new category only if you already have 5+ notes that need it.
- You cannot retrieve a recurring type of note fast enough.
- You keep duplicating notes because you cannot find the existing ones.

### Index and MOC patterns for categories
- One category index note per major category (ex: "Decision Logs Index").
- MOCs for active domains or roles (ex: writer, project manager).
- Saved searches for common queues.

### Anti-patterns like over-tagging and deep trees
- Use [[Processing and Organizing Notes]] to apply categories consistently during processing.
- Over-tagging: 8-15 tags per note, most used once.
- Deep folders: 6+ levels where you cannot guess where something lives.

# Knowledge Categorization

Organizing knowledge into categories or tags is crucial for effective retrieval and application. Some methods include:

1. **Tagging**: Assigning keywords to notes for easy search and retrieval.
2. **Folders**: Grouping related notes into hierarchical folders.
3. **Linked Notes**: Creating connections between related notes using hyperlinks or back-links.
4. **Indexing**: Creating an index of topics or ideas for quick reference.
