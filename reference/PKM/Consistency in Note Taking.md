> If notes are hard to query and sort, this note shows the few metadata fields that matter, without turning every note into a form.

### What to standardize (minimal set)
- A concept note always has a short definition and one example.
- Titles: stable and search-friendly (see [[Note Naming Conventions]]).
- Frontmatter fields: the few you will actually query (see [[Consistency in Note Taking]]).

### Metadata fields to standardize
- WHAT: what the note is for (human quick scan).
- aliases: what you might search.
- tags: small grouping labels from the canon.

### Naming and metadata rules that matter most
- Tags are used sparingly, from the canon, and never as a substitute for links (see [[Tag Canon]]).
- Titles match what you would search.
- Aliases cover common synonyms.
- Acronyms: "PKM", "GTD", "ZK".
- Common phrasing: "Meeting Minutes" vs "Meeting Notes".
- Synonyms: "Findability" vs "Retrieval".

### Tag CSV rules and minimal schema
- Tags are for grouping, not relationships (see [[Tag Canon]]).
- Use as few tags as possible (0-2 typical, 0-4 max).
- Only use canonical tags from [[Tag Canon]].

### Status fields for lifecycle and reviews
- draft notes get processed during [[PKM Daily Review]].
- draft: captured but not distilled.
- stable: good enough to rely on.

### Review routines to enforce consistency
- Daily: fix one obvious inconsistency during [[PKM Daily Review]] (rename, add missing alias, add one link).
- Weekly: run one query for "missing metadata" and fix 3 notes.
- Monthly: prune templates and consolidate tags.

### How to measure and adjust standards
- Retrieval is fast and reliable.
- New notes feel easy to create.
- You refactor less often because notes start cleaner.
