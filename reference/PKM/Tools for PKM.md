> [!summary]
> Select a tool stack for capture, writing, sync, search, and backup without lock-in.

> [!When to use]
> If tool choice is overwhelming, this note helps you pick a durable PKM stack that fits your constraints, without lock-in.

### Core tool jobs (capture, edit, search, sync, backup)
- Capture: fast intake on every device.
- Edit: comfortable writing and structure.
- Search: reliable retrieval (titles, text, tags, metadata).

### Selection criteria (speed, portability, cost)
- Choosing tools for novelty, not for reliability. Fix: optimize for boring, dependable workflows.
- [[Local First File Over App]] for ownership and portability.
- Speed: can you capture and search in seconds?

### Minimal viable stack vs advanced stack
- One capture inbox (mobile + desktop).
- One Markdown vault editor.
- One sync method (optional).

### Integration and migration considerations
- Building a stack where one tool is the single point of failure. Fix: keep a manual fallback and export path.
- Can you move content in and out as plain text?
- Are your metadata fields portable?

### Rules to prevent tool bloat
- One-in, one-out: adding a tool requires removing or de-emphasizing another.
- Make output the scoreboard: tools must increase shipped artifacts.
- Default to plain files.
