> [!summary]
> Organize by Projects, Areas, References, Archives to keep material actionable.

> [!When to use]
> If everything is everywhere, PARA gives you four buckets that match action and context; this note shows how to set it up without deep folder trees.

### PARA categories and definitions
- **Projects**: outcomes with an end date (launch, paper submission, migration).
- **Areas**: ongoing responsibilities (health, finances, team leadership).
- **References**: reference material and reusable knowledge (methods, templates, concepts).

### Rules for moving items across buckets
- When a project ends, move it to Archives.
- When an area becomes time-bound, it becomes a project.
- When a reference becomes active work, it becomes a project (or links from one).

### How PARA interacts with linking and MOCs
- Link reference notes into multiple projects when reused.
- Maintain a project hub note that links to meetings, decisions, and references.
- Use MOCs for topics that span multiple PARA buckets.

### Failure modes like stuffing References
- Fix: enforce a relevance filter (see [[Focus on Relevance]]). If you cannot name a likely future use, do not keep it.
- Fix: move only at clear transitions (project start/end), and rely on links for everything else.
- Fix: rewrite the item as an outcome with an end date, or move it to Areas.
