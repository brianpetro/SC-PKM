# Smart Connections scene - 07 Bases related links column

## outcome

```md
Base row -> list_connections formula -> related-link trail
```

## user belief

> Connections Pro can make semantic relationships actionable inside a Base by showing related note links for each row.

## publication eligibility

| Context | Eligibility | Constraint |
|---|---|---|
| Smart Connections Core conversion | no | Bases integration is Pro |
| Smart Plugins ecosystem | yes | label Connections Pro |
| Connections Pro conversion/docs | yes | formula output must be real current product output |
| Procedural documentation | yes | formula setup may be a separate asset |
| QA evidence | yes | record formula, Base state, and plugin versions |

## truth contract

- The related-link column uses the current documented Connections Pro `list_connections` Bases function.
- Links are generated by the current plugin for real fixture rows.
- The still proves populated related-link trails inside a Base.
- It does not prove a guaranteed number of links, universal ranking quality, or large-vault speed.
- The scene must not present Bases integration as Core.

## prerequisites

### profile

- Restore `sc-pro-clean`.
- Enable Smart Environment, Smart Connections Core, and Connections Pro.
- Disable Smart Lookup and Smart Context.
- Confirm indexing is complete.

### fixture

- Restore `smart-connections-demo-v2` at a verified `v2-bases-scored` state produced by Scene 05.
- Confirm `Smart Connections Relevance.base` is active.
- Confirm candidate rows load and the score column contains no blank or error values.

## start state

- `Smart Connections Relevance.base` is open in the main workspace.
- The current score column may remain visible.
- No formula/property editor, command palette, filter editor, or menu is open.
- No `list_connections` column exists yet in the clean start state.

## access actions

1. Open the Base's current Properties or formula-column interface.
2. Add a formula property using the current Bases UI.
3. Name the visible column `Related connections`, unless the current UI generates a documented default name.
4. Enter the documented formula through the formula editor:

```text
file.list_connections()
```

5. Confirm the formula is accepted without an error.
6. Make the new column visible in the current table view.
7. Resize columns through normal Bases controls so note titles and related links are readable.
8. Keep 5-8 meaningful rows visible.
9. Wait until every visible formula value is stable.
10. Confirm the result contract before capture.

Do not hand-edit undocumented `.base` syntax. Use the current Bases UI and documented formula function.

Record the exact current Properties/formula labels, formula autocomplete behavior, generated property identifier, and rendering behavior in the first verification run.

## stop state

The scene is ready only when:

- Base/table identity is visible
- note-title rows are visible
- `Related connections` or the current verified generated column is visible
- at least 4 visible rows have populated real related-link trails
- links are readable and not clipped into unusable fragments
- no formula editor, menu, loading, blank, or error state covers the evidence

## result contract

- At least 4 visible related-note rows have one or more generated links.
- At least 2 visible link trails point to another fixture note from `Smart Connections Demo/Related/`.
- No visible formula cell contains an error, `NaN`, unresolved token, or loading placeholder.
- All links are current plugin output and remain unedited.
- The exact number and order of links remain whatever the current product returns.

## must show

- Base identity
- note-title rows
- related-link formula column
- at least 4 populated cells
- enough surrounding table context to make the row-to-links relationship obvious

## must not show

- Core-only labeling or implication
- manually typed note links presented as formula output
- formula errors or loading states
- command palette, setup modal, or property editor in the primary final state
- clipped cells that make link trails unreadable
- a caption claiming large-vault speed or universal semantic correctness

## recovery

| Symptom | Minimum recovery | Escalate when |
|---|---|---|
| Formula function is absent | Confirm Connections Pro is enabled, the Base is active, and current plugin functions are loaded. | documented function remains unavailable in a clean current build |
| Formula is rejected | Re-enter the documented instance form `file.list_connections()` through the current UI. | the current function signature differs materially |
| Cells are blank/loading | Confirm indexing, wait for refresh, then reopen the Base once. | values remain blank after a clean indexed restore |
| Links are unreadably dense | Widen the column and reduce visible rows through normal view controls. | no truthful readable table composition is possible |
| Negative-control rows dominate | Filter the presentation view to the fixture Related folder through current Bases controls and record the filter. | only manual data editing can produce a coherent scene |
| Duplicate formula columns exist | Restore `v2-bases-scored` instead of deleting ad hoc during the official run. | clean snapshot already contains duplicates |
| Pro ownership is ambiguous | Stop and restore `sc-pro-clean`; use a Pro caption in publication. | ownership cannot be established |
